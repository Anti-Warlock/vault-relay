# 为什么长内容需要更好的发布工具

短内容可以直接在社交平台的编辑框中完成，但一篇结构完整的长内容通常需要反复调整段落、拆分观点、检查长度，并重新安排图片的位置。真正耗费时间的不是点击发布，而是把已经写好的内容重新整理成目标平台适合阅读的形态。

对于习惯 Markdown 的用户来说，Obsidian 是内容的原始来源。理想的工作流应该保留原始文章，并根据不同平台的限制生成适合发布的版本，而不是让用户在多个编辑器之间重复修改同一篇内容。

## 验证重点

- 长段落是否会按照完整句子拆分
- 列表是否仍然容易阅读
- 每条内容是否控制在平台限制以内
- 最后一条是否不会只剩一小段零散文字
- 图片是否被映射到它附近的正文条目

## 一次编辑，多处使用

用户真正想要的不是简单地把同一段文字复制到所有平台，而是保持核心观点不变，同时让内容符合每个平台的阅读习惯。X 更适合结构清晰的线程，知乎更适合完整文章，小红书则更依赖图片和简洁的段落。

这意味着产品的核心价值应该是内容转换与发布辅助，而不是模拟点击。发布动作本身没有太高价值，减少重复排版、图片整理和内容拆分所花费的时间才有价值。

## 最终检查

发布前仍然应该由用户检查内容。自动转换可以减少机械工作，但不能替代作者判断语气、上下文和图片是否合适。一个可靠的工具应当帮助用户更快完成检查，而不是隐藏发布过程。
